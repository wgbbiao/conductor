# 回放（Replay）功能设计

> 状态：Draft · 负责人：Conductor Contributors
> 关联：[ADR-0002 事实源与事件账本](../adr/0002-source-of-truth-and-event-ledger.md) · [功能模块全景 B10](./modules-overview.md) · [Phase 1 计划](../plans/2026-07-09-phase-01-backend-core.md)
>
> 背景问题：README 与 ROADMAP 都宣传"可回放"，`AuditEvent`/`ToolEvent` 事件账本也为此设计，但**"回放"这个功能本身从未被定义**——触发方式？粒度？只读重放还是可重新执行？这是承诺缺口。本设计补上这层定义，并给出分档实现路径。

---

## 1. 为什么必须定义"回放"

"可回放"是 Conductor 的核心承诺之一（与"可观测、可治理"并列）。但"回放"在不同语境下指完全不同的东西：

- **只看历史**：像看录像，把发生过的事按时间重看一遍。
- **重新执行**：像版本控制，能"回到某一点，从那里换个方式继续"。

这两者的实现成本相差一个数量级。不做定义就宣传"可回放"，等于空头承诺。本设计把它拆成明确的三档，让后续实现有清晰的目标。

## 2. 语义分档（核心）

| 档 | 名称 | 能力 | 实现成本 | 依赖 |
|----|------|------|---------|------|
| **R1** | 时间线回放（只读） | 按时间顺序重看一个 WorkItem / ToolRun 发生的所有事件，像看录像 | 低 | 已具备（事件账本） |
| **R2** | 事件流导出/重建 | 把事件账本导出为标准格式（JSON/NDJSON），可离线分析、可在另一实例重建状态 | 中 | R1 + 导出格式 |
| **R3** | 可重放执行（时间旅行） | 回到某个历史状态点，从那里用不同参数/不同工具重新执行 | 高 | R1 + 状态快照 + 幂等重跑 |

### R1 — 时间线回放（只读）

**定义**：给定一个 `WorkItem`（或 `ToolRun`），按时间顺序返回它的全部事件流：状态流转（来自 `AuditEvent`）+ AI 工具的流式输出（来自 `ToolEvent`）。前端按 `seq`/`createdAt` 顺序渲染，像看录像。

**触发方式**：
- API：`GET /work-items/:id/timeline`（聚合 AuditEvent + 该 item 下所有 ToolRun 的 ToolEvent，按时间排序）
- UI：WorkItem 详情页一个"回放"按钮，进入只读时间线视图。

**为什么是 R1**：事件已经全部持久化（`ToolEvent.seq` 单调、`payload` 完整、`@@unique([runId,seq])` 防重），R1 只是**读取并排序展示**，无需新机制。这是兑现"可回放"承诺的最低门槛。

### R2 — 事件流导出/重建（中）

**定义**：在 R1 基础上，能把事件流导出为标准格式，脱离 Conductor 也能分析；或在另一个 Conductor 实例从事件流重建状态（用于迁移/灾备）。

**额外能力**：
- `GET /work-items/:id/timeline?format=ndjson` 流式导出。
- 一个 `replay-state` 工具：吃进事件流，重放出最终状态（不重新执行 AI，只还原状态机结果）。

**为什么是 R2**：需要定义导出格式规范、并写一个"从事件重建状态"的纯函数（可放 `packages/core`，与状态机规则同层）。

### R3 — 可重放执行 / 时间旅行（高）

**定义**：回到某个历史状态点，从那里用不同参数或不同工具**重新执行** AI 工具，产生新的分支结果。类似 Git 的 checkout + 新 commit。

**为什么是 R3（贵）**：
- 需要状态快照（在每次状态转移时存一份状态快照，否则只能从原始事件从头重放还原中间态——慢）。
- 需要"从某点重新执行"的执行模型：ToolRun 已有 `idempotencyKey` 支持重试，但"分支重跑"会产生**新的 ToolRun**（新 id、新事件流），需要 WorkItem 支持"多版本执行历史"。
- 与现有 `WorkflowRun`（P1 与 WorkItem 1:1）冲突——R3 需要 WorkItem:WorkflowRun = 1:N（一个 WorkItem 多条执行轨迹）。**这是 R3 的前置架构变更。**

---

## 3. 建议：先做 R1，承诺只到 R1

**决策建议**：

- **P1 闭环跑通后，实现 R1**（时间线回放）。它成本低、数据已具备，且能真正兑现"可回放"承诺。
- **README/ROADMAP 的措辞明确为"时间线回放（只读）"**，不宣传 R3 的时间旅行能力，直到真正实现。避免承诺缺口。
- R2 作为 P1 之后的中期目标（可观测性增强）。
- **R3 不承诺时间表**，因为它是架构级的（需要 1:N 执行历史 + 状态快照），应单独起一个 ADR 评估是否值得做。

## 4. R1 的最小实现接口（供后续 plan 参考）

> 这是接口草图，不是执行计划。具体 Task 分解待 R1 正式排期时做。

**后端**：
- `GET /work-items/:id/timeline`：返回 `{ events: TimelineEvent[] }`，按 `ts` 升序。
- `TimelineEvent`：统一 AuditEvent 与 ToolEvent 的投影——`{ ts, source: "system"|"tool", type, summary, detail }`。

**前端**：
- WorkItem 详情页"回放"按钮 → 时间线视图：纵向列表，每条事件一行，左侧时间轴，区分系统事件（状态流转）与工具事件（AI 输出）。

**数据**（无需改 schema）：
- AuditEvent 按 `(subjectType="WorkItem", subjectId)` + `createdAt` 取。
- ToolEvent 通过 WorkItem → ToolRun[] → ToolEvent 取，按 `seq` 排序。
- 两者合并按全局 `ts` 排序。

## 5. 风险与边界

- **事件量大**：长任务 ToolEvent 可能很多。R1 的 timeline 接口应分页/流式，避免一次拉取海量事件。
- **R3 的 WorkItem:WorkflowRun 关系**：现有 P1 设计是 1:1，R3 需要 1:N。若未来确要做 R3，`WorkflowRun` 模型需扩展（保留"主轨迹"与"实验分支"）。这不在本设计范围内，但标记为已知约束。
- **回放≠撤销**：R1 是只读的，不改变当前状态。R3 的"重跑"产生新分支，也不抹除历史。本系统不做"撤销已发生事件"——审计账本不可篡改是 ADR-0002 的核心承诺。
