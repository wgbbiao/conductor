# 🗺️ Roadmap

> Conductor 是一个 AI 原生的研发流程编排平台。本文记录其演进路线。
> 状态图例：✅ 已完成 · 🚧 进行中 · 📋 规划中 · 💡 构想中

---

## Phase 0 — 立项 (Incubation) 🚧

**目标**：明确项目定位、范围与边界，建立社区共识。

- [x] 确立项目定位：AI 原生的研发流程编排平台
- [x] 立项文档（README / LICENSE / CONTRIBUTING / COC）
- [ ] 建立核心设计文档与 ADR 记录机制
- [ ] 明确与同类项目（Linear / Jira / Cursor / Devin）的差异化边界

---

## Phase 1 — 架构骨架 📋

**目标**：搭起核心领域模型与扩展点，让"可插拔"成为一等公民。

- [ ] 领域模型：项目 / 迭代 / 工作项 / 角色 / 工作流
- [ ] 三个扩展点抽象：`ToolProvider`（AI 工具）/ `SkillPack`（技能包）/ `Role`（角色）
- [ ] 最小可运行内核 + 插件注册机制
- [ ] 技术栈选型与 ADR 定稿

---

## Phase 2 — AI 工具适配层 📋

**目标**：让真实 AI 编码工具可以接入并被调度。

- [ ] 适配 Codex
- [ ] 适配 Claude Code
- [ ] 工具调用协议抽象（统一接口，工具可热替换）
- [ ] 调用过程记录与回放

---

## Phase 3 — Skills 挂载机制 📋

**目标**：创建项目时可挂载不同 skills 生态。

- [ ] Skill 包的描述规范与解析
- [ ] 适配 OpenSpec
- [ ] 适配 Superpowers
- [ ] 适配 gstack
- [ ] skills 组合与冲突处理

---

## Phase 4 — 多角色工作流 📋

**目标**：不同角色在同一流程中协作。

- [ ] 角色与权限模型（PM / 开发 / 测试 / 设计 / Reviewer）
- [ ] 工作流状态机（需求 → 设计 → 编码 → 测试 → 发布）
- [ ] 人机交接点（Hand-off）的显式建模

---

## Phase 5 — 全流程贯通 📋

**目标**：端到端研发流程在 Conductor 内贯通、可观测、可治理。

- [ ] 端到端流程编排引擎
- [ ] 全流程可观测看板
- [ ] 审计与回放
- [ ] 与外部系统（Git / CI）的集成

---

## 长期愿景 💡

- 成为 **AI 时代研发流程的"操作系统"**
- 形成开放的 skills / 工具适配生态
- 沉淀可复用的"人机协同研发"最佳实践

---

> 路线图会随社区讨论持续演进，欢迎在 Issue / Discussion 中提出你的诉求与优先级建议。
